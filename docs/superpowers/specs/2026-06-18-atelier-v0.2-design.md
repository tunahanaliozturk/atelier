# Atelier v0.2 — Enhancement Design

Date: 2026-06-18

## Goal

Grow Atelier from a working orchestration skeleton into a usable crew. The orchestration
machinery (lead, planner, capability routing, mission tracking, dispatch, verify) already
exists, but there are no implementation agents to route work to: every current agent sits
in the `orchestration` or `review` layer, so the lead can plan and review but cannot
dispatch real work. This update fills that gap and rounds out the surrounding workflow.

Four phases, ordered by value, shipped together as v0.2.

## Non-goals

- No change to the markdown-driven architecture (agents/commands/skills stay as `.md`
  instruction files; `lib/` holds small, tested `.mjs` helpers).
- No new runtime or executable orchestrator; Claude still performs orchestration by
  following skill instructions.
- No stack-specific assumptions baked into agents (the crew is stack-agnostic).

## Phase 1 — Specialist crew

Add implementation agents so the capability filter has real candidates in the stack
layers. Each agent is a new file under `agents/` following
`docs/schemas/capability-schema.md`. The `SessionStart` hook surfaces them automatically;
no `lib` change.

| Agent | layer | task_kinds | Role |
| --- | --- | --- | --- |
| backend-engineer | backend | implement, refactor, test | APIs, services, business logic; writes its own tests |
| frontend-engineer | frontend | implement, refactor, test | UI, components, styling, state |
| infra-engineer | infra | implement, configure, automate | CI/CD, containers, IaC, deploy |
| data-engineer | data | implement, migrate, model | schema, migrations, queries, data pipelines |
| docs-writer | docs | document | README, API docs, guides |
| security-reviewer | review | security-review | cross-cutting; lead calls it by name like the other reviewers |

Design decisions:

1. **Tests live inside the layer.** Rather than a dedicated test-engineer plus a new
   `quality` layer, each stack engineer writes its own tests (`task_kinds` includes
   `test`). This keeps the schema unchanged and matches the planner's "small,
   independently testable steps" philosophy.
2. **security-reviewer sits in the `review` layer.** The schema defines `review` as
   cross-cutting (reviewers are called by name in a fixed order, not selected by the
   coarse filter). Giving it `task_kinds: [security-review]` keeps the registry
   unambiguous and avoids a schema change.

After this phase the coarse filter finds real candidates across all five stack layers, so
the "no agent fits" path becomes the exception rather than the norm.

Touches: six new files under `agents/`. No changes to `lib`, the hook, or existing agents.

## Phase 2 — Routing intelligence

Today routing is purely a skill instruction: the lead reads `description`s and picks
heuristically, which is neither reproducible nor auditable. This phase moves the
mechanical part into a deterministic, tested helper that the skill leans on.

New `lib/routing.mjs` (consumes the output of `registry.mjs`):

- `coarseFilter(task, agents)` — keep agents whose `layer` fits the task and whose
  `task_kinds` include the task's kind (the code form of the skill's stage 1).
- `scoreCandidate(task, agent)` — numeric score from `task_kinds` overlap plus overlap of
  `capabilities` with the task's required tags. Pure, side-effect free.
- `route(task, agents)` → `{ candidates, pick, reason, fallback }`. Candidates ranked by
  score descending, then name ascending (stable, deterministic tie-break). When the coarse
  filter is empty, `pick = null` and `fallback = true`.

Routing input is a lightweight task descriptor `{ kind, layer?, capabilities? }` — the
tags the planner already produces. The mission table is unchanged; routing runs at
decomposition time (when assigning agents), not from a mission row.

New `lib/routing.test.mjs` — exact match, multi-candidate ranking, deterministic
tie-break, and no-fit fallback.

Updated `skills/capability-routing/SKILL.md` — the skill now leans on the helper: coarse
filter → score → deterministic tie-break → record the choice with *score and reason* in
the decision log. The fallback behavior (no fitting agent → ask the user) is unchanged but
is now driven by `route()`'s `fallback` flag.

Design decision: the helper **strengthens, not replaces** the skill. The heuristic nuance
("read the description, pick the closest") remains; the coarse filter and scoring move into
code so routing is reproducible and the decision log is auditable by score.

Touches: `lib/routing.mjs` (new), `lib/routing.test.mjs` (new),
`skills/capability-routing/SKILL.md` (update). Registry, mission, and hook unchanged.

## Phase 3 — New commands and flows

The mission file persists at `.atelier/missions/<date>-<goal-slug>.md`, so a run can be
continued in a new session. These commands build on that.

New commands (each a `commands/*.md`):

- **`/resume`** — read the most recent mission file (`parseMission`), find tasks that are
  not `done`, recompute the runnable wave (deps satisfied), and continue dispatch from
  where it left off. The plan is already approved, so there is no new approval gate; the
  command shows the board and asks for a "continue?" confirmation so dispatch is never a
  surprise. For picking up after an interruption or in a fresh session.
- **`/retry <id>`** — re-run a `blocked` or failed task: reset its status, append a
  decision-log entry, and send it back through the verify loop (spec review → quality
  review). With no argument, `/retry` retries all `blocked` tasks.
- **`/crew`** — show the capability registry (agents, layers, capabilities, task_kinds)
  via `formatRegistry` on demand. `/status` shows the mission board; `/crew` shows the
  crew board.

New skill `writing-agents` — the scaffold promised in the capability-routing fallback
("arrives in a later release"). For a task no agent covers, it guides creating a new agent
`.md` with schema-correct frontmatter and body; the `SessionStart` hook then surfaces it
automatically. The routing fallback now points at a real path.

Small update: the "arrives in a later release" note in `capability-routing/SKILL.md` is
replaced with a link to the `writing-agents` skill.

Design decision: the parse and runnable-wave logic already lives in `lib/mission.mjs`; the
commands *use* it rather than re-implementing it. The status-reset helper `/retry` needs is
specified in Phase 4 (`setTaskStatus`).

Touches: `commands/resume.md`, `commands/retry.md`, `commands/crew.md` (new);
`skills/writing-agents/SKILL.md` (new); `skills/capability-routing/SKILL.md` (small edit).

## Phase 4 — Quality and robustness

The hook is already best-effort (a registry load failure never breaks the session). This
phase tightens validation, tests the new helpers, and aligns the docs.

`lib/mission.mjs` — validation plus helper:

- `validateMission(mission)`: status values are in the enum
  (`pending/in-progress/blocked/done`); every `deps` entry references an existing task id;
  no dependency cycles; no duplicate task ids. Failures throw with a clear message.
- `setTaskStatus(mission, id, status)`: the single, validated path for status transitions
  used by `/retry` and by dispatch. (This is the status-reset helper referenced in
  Phase 3.)

`lib/registry.mjs` — validation:

- `layer` is in the allowed enum
  (`orchestration/backend/frontend/infra/data/docs/review`); `task_kinds` is non-empty; no
  duplicate agent names. The hook catches these and reports the offending file by name
  (the existing graceful behavior is preserved).

Tests: new validation scenarios in `mission.test.mjs` and `registry.test.mjs`; Phase 2's
`routing.test.mjs` already exists. `node --test` must stay green.

Documentation alignment:

- `README.md`: new commands (`/resume`, `/retry`, `/crew`) and the populated crew.
- `CHANGELOG.md`: a v0.2 entry under `[Unreleased]` (crew, routing helper, new commands,
  writing-agents skill, validations).
- `docs/schemas/capability-schema.md`: reflect that the stack layers are now populated.
- `docs/schemas/mission-schema.md`: a short reference to the `validateMission` rules (the
  status enum is already documented).

Design decision: all validation lives in pure functions in `lib`; skills and commands call
them. Rules stay in one place, testable, and consistent across the hook and commands.

Touches: `lib/mission.mjs`, `lib/registry.mjs`, `lib/mission.test.mjs`,
`lib/registry.test.mjs` (update); `README.md`, `CHANGELOG.md`, both schema docs (update).

## Testing strategy

- Every new `lib` helper ships with `node --test` coverage (`routing.test.mjs` new;
  validation cases added to `mission.test.mjs` and `registry.test.mjs`).
- Agents, commands, and skills are instruction files verified by review against their
  schemas, not by automated tests.
- Acceptance: `node --test` green; the hook still surfaces the registry (now with the full
  crew); `/orchestrate` can route an implementation task to a stack engineer end to end.

## Rollout

Single v0.2, phases implemented in order (1 → 4) so each builds on the last: the crew
exists before routing scores it, routing exists before the commands drive it, and
validation hardens all of it last.
